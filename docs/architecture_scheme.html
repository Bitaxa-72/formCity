Главная дорога

Это нормальный путь текстового запроса к данным. Команды и ошибки уходят в отдельные ветки.

01

Telegram Update

Бот получает update из Telegram polling или webhook.

{
  "message": {
    "from": {"username": "vitalyfrontend"},
    "chat": {"id": 885428167},
    "text": "платежный календарь март итоги"
  }
}
↓
02

Access

Проверяется username по allowlist. User id сейчас не используется.

В списке - идем дальше
Не в списке - отправляем отказ

03

Command Router

Если текст начинается с команды, LLM не вызывается.

  • /start - приветствие и сброс рабочего контекста
  • /info - справка
  • /clear - сброс рабочего контекста
  • /admin - включить или выключить debug
↓
04

User Session

Загружается пользователь, dialog_state, history и last_result.

{
  "dialog_state": {
    "report_type": "payment_calendar",
    "project": "all",
    "period": {"from": "2026-03-01", "to": "2026-03-31", "label": "март"},
    "metrics": ["plan", "fact", "deviation"],
    "filters": {},
    "group_by": ["project"]
  }
}
↓
05

LLMInput

Backend собирает пакет для LLM: сообщение, текущий контекст, историю, last_result и dictionary.

{
  "user_message": "факт по рекламе",
  "dialog_state": {...},
  "dictionary": {
    "intent": ["data_query", "dimension_query", "..."],
    "report_type": ["payment_calendar", "..."],
    "metric": ["plan", "fact", "deviation"],
    "report_rules": {"payment_calendar": {...}}
  }
}
↓
06

LLMParsedResponse

LLM возвращает строгий JSON. Backend валидирует schema и делает repair-pass при ошибке.

{
  "intent": "data_query",
  "state_delta": {
    "metrics": ["fact"],
    "filters": {"article": "реклама"}
  },
  "operation": null,
  "needs_clarification": false,
  "confidence": 0.9
}
↓
07

Context Resolver

Применяет state_delta к dialog_state и решает: новый запрос, продолжение или уточнение.

{
  "report_type": "payment_calendar",
  "project": "all",
  "period": {"from": "2026-03-01", "to": "2026-03-31", "label": "март"},
  "metrics": ["fact"],
  "filters": {"article": "реклама"},
  "group_by": ["project"],
  "last_intent": "data_query"
}
↓
08

QueryFrame

Приводит контекст к рабочему запросу и проверяет обязательные поля.

{
  "ready": true,
  "report_type": "payment_calendar",
  "project": "all",
  "metrics": ["fact"],
  "filters": {"article": "реклама"},
  "group_by": ["project"],
  "missing_fields": []
}
↓
09

Domain Resolver

Сверяет свободные значения с БД. Например, article="реклама" превращается в точное article="Реклама".

{
  "valid": true,
  "frame": {
    "filters": {"article": "Реклама"}
  },
  "errors": []
}
↓
10

Metric Resolution

Проверяет, что метрики, фильтры и группировки разрешены для выбранного типа отчета.

{
  "valid": true,
  "metrics": [
    {"name": "fact", "unit": "rub"}
  ],
  "errors": []
}
↓
11

SQLQuery

SQL собирается только из whitelist таблиц, колонок, метрик и фильтров. LLM SQL не пишет.

SELECT
  project AS project,
  SUM(fact_amount) AS fact,
  SUM(plan_amount) AS plan,
  SUM(deviation_amount) AS deviation,
  COUNT(*) AS source_rows
FROM payment_calendar_facts
WHERE period_month >= :date_from
  AND period_month <= :date_to
  AND article = :filter_article
GROUP BY project
↓

CalculationResult

Выполняется SQL или операция по last_result.

{
  "kind": "sql_result",
  "rows": [
    {"project": "moskovsky", "fact": 1686810, "plan": 2540000, "deviation": -853190}
  ],
  "row_count": 1,
  "metrics": ["fact"]

13

Result Verification

Проверяет пустой результат, отсутствующие метрики и источник данных.

↓
14

Готовит чистый JSON для ответа. Внутренние колонки, например source_rows, скрываются.

Развилки

Главные условия, которые меняют дорогу запроса.

Команда

Если сообщение является командой, LLM не вызывается.

/start
-> сброс рабочего контекста
-> приветствие

/admin
-> переключить debug
-> сохранить флаг в state

Нет доступа

Если username не в allowlist, бот сразу отправляет отказ.

{
  "access": "denied",
  "message": "У вас нету доступа..."
}

Health failed

Если Telegram, LLM API или proxy недоступны, запрос не идет дальше.

{
  "health": "failed",
  "failed_service": "llm"
}

General question

Приветствие или общий вопрос не меняет рабочий контекст.

{
  "intent": "general_question",
  "state_delta": {}
}

Unsupported

Запрос вне отчетов не должен запускать SQL.

{
  "intent": "unsupported",
  "state_delta": {}
}

Query not ready

Если не хватает report_type, metrics или dimension, бот задает уточнение.

{
  "ready": false,
  "missing_fields": ["metrics"],
  "clarification_question": "Уточните метрику..."
}

Domain error

Если статья не найдена или совпадений слишком много, SQL не запускается.

{
  "valid": false,
  "errors": ["article_ambiguous"],
  "clarification_question": "Уточните статью..."
}

Metric error

Если метрика не относится к отчету или фильтр запрещен, нужен вопрос пользователю.

{
  "valid": false,
  "errors": ["metric_not_allowed_for_report_type"]
}

Правила контекста

Контекст хранится как набор полей. Главное правило: не терять уже понятный смысл и не наследовать опасные поля без причины.

Ситуация
Что делаем
Что сохраняем
Новый data_query с report_type
Начинаем новое дерево
Только поля из нового запроса
data_query без report_type, но старый report_type есть
Продолжаем прошлый контекст
report_type, period, project, если они не изменены
Бот ждет уточнение
Мержим ответ поверх clarification_base_state
Все старые поля, если пользователь их не менял
Уточнение report_type
Дополняем контекст
metrics, filters, period не чистим
Частичный запрос без metrics в payment_calendar
Ставим базовые metrics
metrics = plan, fact, deviation
project = all
Не смешиваем проекты
Добавляем group_by = project
